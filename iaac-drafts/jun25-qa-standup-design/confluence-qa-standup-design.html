Purpose

This page captures the design and acceptance criteria for standing up the op-usxpress-qa on-prem Talos cluster, with explicit production-readiness hardening that closes the gaps identified in our Dev environment retrospective. The intent is that QA is not a relaxed copy of Dev — it is the first cluster built to the standard Prod will be held to. Dev remains the experimentation surface; QA is the integration + acceptance environment that Prod will mirror.

This design follows industry best practice for production-like Kubernetes environments. Every section below addresses a category of concern that any production-bound cluster must explicitly answer; where Dev took shortcuts, QA does not.

Source of best practice

The structure below reflects the consensus pattern across mature on-prem Kubernetes operators (large financial services, regulated SaaS, defense-adjacent enterprises). Each section maps to a category of concern that production governance frameworks (CIS Kubernetes Benchmark, NIST 800-190, SOC 2 CC sections) hold operators to. We document our specific implementation rather than the abstract control; the abstract controls are inherited from those references.

1. Per-environment account and secret boundary

Each environment has explicit isolation of the off-cluster resources it depends on. The boundary is enforced at the AWS account and secret-path level, not at the Kubernetes namespace level alone.

Rationale

A per-environment AWS account boundary is the most defensible isolation we can give the on-prem clusters. A compromised Dev secret cannot be used to read Prod state; a Dev IAM role cannot list Prod buckets. The cost of the per-account boundary is an additional set of AWS resources to provision and a path-to-promote for resources that are intentionally identical across environments (ECR being the deliberate exception — we want the same image to be promotable from Dev to Prod).

Implementation

  • The QA state bucket is provisioned in USX-QA at QA cluster-creation time, following the same lazy-tf-state naming convention
  • S3 CRR enabled at provisioning time, not retrofitted (carry the SSE-KMS triple-gotcha checklist from the Dev work)
  • AWS SM path prefix op-usxpress-qa/ is the only path the QA cluster's IRSA roles can read; enforced via IAM policy condition
  • External-Secrets ClusterSecretStore on QA uses an IRSA role that only allows op-usxpress-qa/* access

2. Capacity planning and sizing

Dev was sized for experimentation: three control plane nodes at 8 GB each plus seven worker nodes at 4 GB each, with no node-pool isolation. QA is sized for production-like workload patterns: control plane appropriately sized for etcd and API throughput, and worker pool divided across three node classes (see Section 3 for the node pool design).

Proposed QA sizing

System worker pool248 GB50 GB SSD—kube-system DaemonSets, CNI, kube-proxy, CoreDNS — small, dedicated, never co-scheduled with platform or app workloads

Total resource budget

  • Control plane: 12 vCPU, 48 GB RAM, 300 GB storage
  • System worker pool: 8 vCPU, 16 GB RAM, 100 GB storage
  • Platform worker pool: 24 vCPU, 48 GB RAM, 300 GB storage
  • Application worker pool: 80 vCPU, 160 GB RAM, 1500 GB storage
  • Total: 124 vCPU, 272 GB RAM, 2.2 TB storage

Rationale

Dev's 4 GB control plane nodes were destroyed by the Rook CSI OOM cascade on 2026-06-17 because Rook CSI driver pods land on every node. We raised Dev CP to 8 GB but that was the minimum to recover, not the target for a production-like environment. QA gets 16 GB CP nodes from day one with a separate disk for etcd (etcd write fsync latency is the single most important indicator of API server health). The node pool split (Section 3) means platform and application contention is engineered in, not accidental.

3. Node pool isolation

This is the most important architectural decision in this design. In Dev, every worker node is "general purpose" — kube-system pods, monitoring, RisingWave, application workloads, and Rook OSDs all schedule onto the same workers. This means a noisy application can starve the platform, a memory leak in monitoring can evict an application, and a Rook OSD recovery can spike CPU on a worker hosting tenant traffic. In QA we explicitly separate three classes of workload onto dedicated node pools.

The three pools

PoolLabelTaintWhat runs hereWhat does NOT run here
System node.cloud-platform.io/pool: system node-role.cloud-platform.io/system:NoSchedule — only system DaemonSets tolerate kube-proxy, CoreDNS, CNI (Cilium agents), kube-system DaemonSets, the small set of cluster-wide infrastructure that must run on every node or in dedicated capacity Cluster-platform services (cert-manager, ESO, Velero, monitoring, Istio control plane) — those belong on the platform pool
Platform node.cloud-platform.io/pool: platform node-role.cloud-platform.io/platform:NoSchedule — only platform workloads tolerate cert-manager, ESO controller, Velero server, Rook operator, Prometheus, Grafana, Loki, Kyverno controller, Reloader, Istio control plane, External-DNS, Flux controllers Tenant applications, RisingWave compute, anything customer-traffic-bearing
Application node.cloud-platform.io/pool: app Untainted (default scheduling target) RisingWave compute and meta workers, tenant applications (geo-handler, brands-api, attrition-api), future tenant onboardings, Rook-Ceph OSDs (only on app pool nodes that hold the secondary data disk) Platform services that need stable capacity; system DaemonSets

How the isolation is enforced

  1. Talos machine configuration labels each worker pool at provisioning time. The Talos config for system workers includes kubelet.extraArgs: { node-labels: "node.cloud-platform.io/pool=system" }; same pattern for platform and app pools.
  2. Talos machine configuration applies the taint at provisioning time for system and platform pools. Application pool is untainted because it is the default scheduling target.
  3. Every platform Helm release includes a nodeSelector and tolerations stanza that selects the platform pool. This is shipped in the Flux Kustomizations under iaac-talos-flux-platform. Existing Dev releases gain these settings as part of the QA rollout so the same chart values apply to both clusters.
  4. Anti-affinity rules ensure RisingWave compute pods, brands-api replicas, and other multi-replica workloads spread across application pool nodes (not co-located on a single host).
  5. Rook-Ceph OSD placement targets only application pool nodes that have the secondary data disk. The CephCluster spec's placement.osd includes the application-pool nodeSelector and an anti-affinity rule across hosts.

Why this matters

Three concrete scenarios this isolation prevents:

  1. A memory leak in Prometheus on Dev would push a worker hard against memory limits and trigger OOMKills on whichever workloads happened to be co-scheduled. Under node pool isolation, Prometheus is constrained to the platform pool; an application pool worker hosting RisingWave cannot be evicted by Prometheus pressure.
  2. A Rook OSD recovery operation spikes CPU and disk I/O. Under Dev's flat pool, that spike can starve a tenant application. Under isolation, OSDs live on application pool nodes with the secondary data disk; tenant applications are co-located but the host has resources sized for both. Platform workloads are completely untouched.
  3. A tenant application gone runaway — an aggressive batch job, a misconfigured queue consumer — can saturate every worker on Dev because there is nowhere safe for platform pods to retreat. Under isolation, the runaway is bounded to the application pool; cert-manager and ESO and monitoring continue functioning on the platform pool.

vSphere fault-domain spread

Each pool is spread across at least three vSphere ESXi hosts using DRS anti-affinity rules. This means a single ESXi host failure cannot remove all system workers (or all platform workers, or all app workers). The exact ESXi host map will be captured in the implementation ticket; the principle is that no pool is single-host-dependent.

4. Identity management

The identity story for the QA cluster has three concerns, each addressed independently: cluster admin access, workload identity for IRSA, and end-user web SSO into platform tools.

4.1 Cluster admin access

Cluster administrators authenticate to the QA API server using X.509 client certificates issued through the existing split-provisioning pattern documented in onprem-cluster-access-runbook. Each administrator's certificate has a name in the certificate subject that maps to a ClusterRoleBinding in the cluster. No shared cluster-admin credentials; every action is attributable.

The QA cluster will issue at least the following administrator certificates at stand-up time:

  • Doke — cluster-admin ClusterRoleBinding (full access for incident response)
  • Idris — cluster-admin for QA stand-up phase; downgrades to namespace-scoped after stand-up matches the existing Dev pattern
  • Tim Preble — namespace-admin on risingwave namespace only (matches Dev pattern)
  • Break-glass certificate — stored in 1Password, used only during incidents; usage logged via the audit pipeline (Section 6)

4.2 Workload identity (IRSA)

Every workload that calls AWS APIs from the cluster uses an IAM Role for ServiceAccount. The OIDC provider for the QA cluster is registered with AWS at cluster stand-up time. Each workload's ServiceAccount is annotated with the role ARN it should assume; AWS STS issues short-lived credentials when the workload presents its projected ServiceAccount token.

The naming convention for QA IRSA roles is:

op-usxpress-qa-<workload>-<purpose>

Examples:

  • op-usxpress-qa-velero-backup — Velero server reading and writing the backup bucket
  • op-usxpress-qa-etcd-backup — etcd snapshot job writing to the snapshot bucket
  • op-usxpress-qa-external-secrets — ESO reading Secrets Manager values under op-usxpress-qa/*
  • op-usxpress-qa-external-dns — External-DNS writing Route 53 records in op-qa.usxpress.io
  • op-usxpress-qa-cert-manager — cert-manager solving DNS-01 challenges in op-qa.usxpress.io
  • op-usxpress-qa-cluster-autoscaler — if cluster autoscaler is adopted on QA, role for vSphere API access

Each role's IAM policy is scoped to the minimum AWS resources the workload requires. No role on QA can read or write resources owned by the Dev or Prod clusters.

4.3 End-user web SSO

This is the Azure AD identity strategy parked under epic INFRA-1559. For QA the operational definition is:

  • Grafana at https://grafana.op-qa.usxpress.io — AAD OIDC SSO via the shared Dev+QA App Registration (per INFRA-1559's recommended per-environment topology).
  • Any future web tools (Kiali, ArgoCD, Loki UI) follow the same App Registration pattern.
  • RBAC mapping — AAD groups arrive as token claims; Grafana role config maps groups to Grafana roles. Initial groups: cloud-platform-admins (Grafana Admin), cloud-platform-engineers (Editor), app-team-readers (Viewer).

5. Log management

Logs from the QA cluster land in a structured pipeline with explicit retention and a search UI. Dev's logs are ad-hoc — kubectl logs works for the moment but there is no long-term destination, no audit trail, no security analytics surface beyond Wiz onboarding which is still in flight. QA introduces the canonical destination and the retention discipline.

5.2 Pipeline architecture

The collection-to-storage path:

  1. fluent-bit DaemonSet on every worker (system, platform, application pools) collects stdout/stderr from every container, parses standard fields, and ships to Loki.
  2. Loki distributor + ingester deployed on the platform pool. Storage backend is S3 (in the QA AWS account) with the boltdb-shipper index. Per-tenant labels enforce tenant isolation between application logs and platform logs.
  3. Grafana Loki datasource provisioned alongside Prometheus. Engineers query logs via Grafana Explore using LogQL.
  4. Kubernetes audit log is enabled via a Talos audit policy that selects RequestResponse for high-value events (token creation, exec, port-forward, Secret reads) and Metadata for routine API calls. The audit log is written to a file on each CP node; fluent-bit on the CP nodes ships it to Loki with a dedicated stream tag.

5.3 Sizing estimate

  • Dev steady-state log volume is approximately 30-50 GB per day across all classes
  • QA, given equivalent workload and additional audit log enablement, is estimated at 80-120 GB per day
  • Prod, with fully onboarded tenant applications, is estimated at 200-300 GB per day
  • Loki ingester sizing for QA: 2 ingester replicas with 2 vCPU and 4 GB RAM each; distributor 1 replica
  • S3 storage cost for QA: roughly 10 GB hot index, 1 TB cold object storage at peak retention — approximately $25-30 per month

5.4 Audit log policy

The Talos audit policy enabled on QA captures:

  • RequestResponse level — full request and response body — for: create on secrets, create on tokenreviews, create/update/delete on clusterrolebindings, create on pods/exec, create on pods/portforward, create/update on networkpolicies
  • Metadata level — method, resource, requester — for routine reads and writes outside the above set
  • None — skip — for: read-only operations from system controllers (kubelet, kube-controller-manager) that would otherwise drown the log in noise

6. Cluster audit logging destination

(Architecture detailed in Section 5.4.) The destination is Grafana Loki running on the platform pool, with S3 backend for cold storage in the QA AWS account. Retention is 30 days hot in Loki, 1 year cold in S3 via Loki's S3-backed index. Search UI is Grafana Explore against the Loki datasource.

Acceptance criteria for the audit pipeline:

  • A kubectl exec command against any pod on QA appears in Loki as an audit event within 30 seconds
  • The exec event includes the requester's identity (X.509 cert subject)
  • Querying "all exec events in the last 24 hours" returns a complete list
  • The audit log volume is monitored; a sudden spike or drop triggers a PrometheusRule alert

7. DR posture

The QA cluster has explicit RTO and RPO targets and a documented restore procedure that is exercised before the cluster is declared production-ready. Dev's DR posture is implicit and unproven beyond the Velero restore exercise of 2026-06-23; QA upgrades this to documented, exercised, and observable.

7.1 Recovery objectives

ScenarioRTO targetRPO targetMechanism
Single worker node loss15 minutes (kubelet rejoin or replacement)0 (no data on local-path; ceph-block survives)Talos auto-rejoin or replacement VM
Single control plane node loss30 minutes0 (etcd quorum from remaining two CPs)Talos CP recovery, etcd member-replace procedure
Two control plane node loss2 hours0 if any CP responds; up to last backup if quorum-lossetcd snapshot restore from S3 (etcd-backup CronJob)
Full cluster loss (all CPs, vSphere host failure)4 hoursUp to 24 hours (etcd snapshot cadence)Rebuild cluster from Talos config, restore etcd snapshot, Flux reconciles platform, Velero restores tenant PVCs
Data centre lossBest-effort, days — no hot standby cluster in another DC at QA scopeUp to 24 hours via the etcd and PVC backups in S3 (replicated to us-west-2)Spin up new cluster in alternate DC, restore from S3 backups; expected to take days, not hours, at QA tier

7.2 What backs up and where

  • Terraform state — lazy-tf-state bucket in USX-QA, replicated to us-west-2 via the same CRR pattern proven on Dev (INFRA-1557)
  • etcd snapshots — daily CronJob; written to S3 in USX-QA; replicated to us-west-2
  • Velero PVC backups — daily for all PVCs; written to S3 in USX-QA; replicated to us-west-2; restore exercise required before sign-off
  • AWS Secrets Manager values — AWS provides intrinsic replication and versioning; no separate backup required
  • Flux Git state — the source of truth lives in variant-inc/iaac-talos-flux-cluster and variant-inc/iaac-talos-flux-platform; GitHub provides the durability

7.3 Restore drill

Before QA is declared production-ready, the following restore drills are executed and documented:

  1. Delete a non-trivial PVC, restore from Velero, confirm pod returns to Ready and data is intact (passes the existing Dev pattern)
  2. Stop etcd on one CP, verify quorum from remaining two, restart CP, verify etcd member rejoin
  3. Simulate full cluster loss by terraform-destroying the cluster, re-provisioning, restoring etcd from snapshot, and verifying Flux brings the platform back. This drill is the most important and is the gate for "QA is production-ready."

8. Multi-fault-domain spread within the data centre

Dev's seven workers may all sit on a single ESXi host or a single rack today; we have not audited the placement and have not enforced anti-affinity at the vSphere level. QA enforces explicit fault-domain spread.

Requirements

  • Audit the current vSphere host topology to identify at least three ESXi hosts and (where applicable) racks
  • Map each pool to a DRS anti-affinity rule that spreads the pool across at least three ESXi hosts
  • Verify the rule by initiating a planned migration of one pool member; observe DRS places it on a host that does not already host another member of the same pool
  • Document the host-to-VM map in the cluster's runbook so on-call engineers know which physical hosts to inspect during an incident

9. Policy baseline (Pod Security Admission + Kyverno)

QA enforces a stricter policy baseline than Dev. Where Dev tolerates baseline PSA on most namespaces and uses Kyverno mostly for label mutation, QA enforces restricted PSA on tenant namespaces and uses Kyverno for verification and blocking, not only mutation.

Namespace tier policy

Namespace classPSA enforceKyverno policies applied
kube-systemprivilegedMinimal — system components require host access
Platform pool namespaces (cert-manager, eso, velero, rook-ceph, monitoring, istio, kyverno, reloader)baselineImage signature verification (cosign), required label set, network policy required
Application namespaces (risingwave, brands-api, attrition-api, future tenants)restrictedAll of the above plus: no privileged escalation, runAsNonRoot required, seccompProfile RuntimeDefault required, no hostPath mounts, image from approved registry list only

Acceptance

  • A pod that attempts to run privileged in an application namespace is rejected at admission with a clear error
  • An image not signed by the trusted cosign keys is rejected at admission
  • A NetworkPolicy is required in every application namespace; deployments without one are rejected after a grace period

10. Image registry promotion and signing

QA images are promoted from Dev images, not built fresh. The image SHA that ran on Dev is the same SHA that runs on QA. This eliminates "works on Dev but not QA" mysteries caused by accidental rebuild differences.

Promotion mechanism

  • Image promoted from Dev to QA by re-tagging in ECR, not by re-building
  • Cosign signature applied at the QA promotion gate — signed by a Cloud Platform team key stored in AWS KMS
  • Kyverno verifyImages policy on QA application namespaces requires the cosign signature; unsigned images are rejected
  • Promotion to Prod follows the same pattern: SHA frozen at Dev, signed at QA, signed again at Prod with a separate key

Approved registry list

  • 064859874041.dkr.ecr.us-east-2.amazonaws.com — the shared DevOps ECR
  • quay.io/<official upstream charts that we vendor unmodified>
  • docker.io/bitnamilegacy/* — Bitnami legacy images for the unavoidable cases
  • No other registry is permitted on QA application namespaces

11. Per-environment DNS subdomain and TLS strategy

QA gets its own subdomain so that DNS-based tooling, certificate scope, and ingress configuration cannot accidentally reach across environments.

DNS

  • Hosted zone op-qa.usxpress.io created in Route 53 within the USX-QA AWS account
  • External-DNS on QA configured with the IRSA role op-usxpress-qa-external-dns scoped to that hosted zone only
  • The corporate DNS resolves op-qa.usxpress.io via NS delegation to Route 53

TLS certificates

  • cert-manager on QA holds a wildcard certificate for *.op-qa.usxpress.io issued by Let's Encrypt via DNS-01 (matching the proven Dev pattern)
  • Per-team certificates follow the pattern documented in onprem-per-team-cert-pattern-may28
  • Certificate renewal is verified during the DR drill (Section 7.3)

12. Platform application onboarding

The reusable platform-app-expose chart (INFRA-1527) currently targets Dev. For QA we generalise the chart to accept the environment as a parameter and emit the appropriate Istio Gateway, VirtualService, ExternalSecret, IRSA annotation, and ServiceMonitor.

A workload onboarded to QA produces a values file that specifies its environment as qa and inherits the rest. The same values file with prod produces the Prod-bound equivalent. This is the single mechanism by which apps move across environments.

14. Acceptance criteria for QA

QA is declared production-ready — meaning Prod stand-up can proceed from this baseline — when every item below is satisfied and documented.

  1. QA Talos cluster is provisioned and reachable; control plane survives a planned single-CP reboot test without API server unavailability beyond 30 seconds
  2. Three worker pools are provisioned with the labels and taints from Section 3; a test pod with no nodeSelector is scheduled to the application pool; a test platform-style pod with the platform tolerations and selector is scheduled to the platform pool; system DaemonSets land on the system pool
  3. Per-environment AWS account boundary is enforced; QA IRSA roles cannot read Dev or Prod resources (tested via aws sts decode-authorization-message)
  4. Per-environment AWS Secrets Manager path prefix is the only path QA workloads can read (tested via a deliberately-attempted cross-env access that is denied)
  5. Identity management: at least three administrator certificates issued; all administrators verified via kubectl auth whoami; Grafana SSO live with at least one AAD group mapping
  6. Log management: a kubectl exec on a QA pod produces an audit event in Loki within 30 seconds; the event includes the requester identity
  7. DR: the Velero restore drill (Section 7.3 item 1) passes; etcd snapshot exists in S3; the full-cluster-loss drill (item 3) is scheduled and held within 30 days of stand-up
  8. Multi-fault-domain spread: each pool is on at least three ESXi hosts; the DRS anti-affinity rule is in place; the host-to-VM map is in the runbook
  9. Policy baseline: a test pod attempting privileged escalation in an application namespace is rejected with a clear error; an unsigned image is rejected
  10. Image promotion: at least one image has been promoted Dev-to-QA via the cosign-signed re-tag mechanism and is running on QA
  11. DNS and TLS: https://grafana.op-qa.usxpress.io serves the live cluster Grafana with a valid Let's Encrypt certificate
  12. Platform app onboarding: at least one application has been onboarded to QA via the generalised platform-app-expose chart and is serving traffic
  13. Path-to-Production: the gates in Section 13 are enforced by branch protection rules and Octopus approval steps; an attempt to push a change without review is rejected
