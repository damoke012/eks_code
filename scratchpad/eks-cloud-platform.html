Ingress & mesh

North-south traffic (external clients ↔ the cluster) enters through the aws-load-balancer-controller (v3.1.0, ALB/NLB) and a pair of Istio ingress gateways (public + internal). East-west traffic (service ↔ service inside the cluster) runs on Istio 1.29.2 with the Istio CNI installed — workload namespaces are labelled istio.io/dataplane-mode: ambient. Monitoring is the full kube-prometheus-stack (Prometheus, Alertmanager → PagerDuty/Slack, Grafana). Flux (flux2 2.18.4) reconciles the platform layer from iaac-eks-manifests; app deploys come through Octopus / mage-runner (§7–8).

Identity — IRSA

Pods get AWS credentials through IAM Roles for Service Accounts. On EKS this is native: the OIDC provider and the pod-identity webhook are supplied by the managed control plane — nothing to vendor (unlike the on-prem Talos build).

Service accountannotated eks.amazonaws.com/role-arn
→
WebhookEKS pod-identity webhook injects token + role env
→
STSAssumeRoleWithWebIdentity
→
Resultshort-lived creds → AWS API calls

OIDC issuer is EKS-provided: https://oidc.eks.us-east-2.amazonaws.com/id/<cluster-id>. Trust-policy subject must be system:serviceaccount:<ns>:<sa> exactly — a wrong namespace is a silent assume failure (debug via CloudTrail).

07

How an app onboards

An app is defined by a spec.yaml in its own repo — the deployment contract. mage-runner (a Go/Mage binary) reads it, validates against JSON schemas, and drives terraform-variant-apps to build the app's infra and deploy its Helm chart.

The contract — spec.yaml

Declares the app type and what infra it needs. mage-runner validates it against terraform-variant-apps/schemas/ and decides which modules to run.

  • common/ — eks-data, role, replicator, auth, buckets, kafka, postgres, dynamodb, mongodb-{cluster,user,privatelink}
  • apps/ — exactly one of api · cron · handler · ui

The unit — Helm chart

One chart per app type, pulled from ECR:

# 064859874041.dkr.ecr.us-east-2
helm-charts/dx-api
helm-charts/dx-cron
helm-charts/dx-handler
helm-charts/dx-ui

Provisions Deployment, Service, ServiceAccount (IRSA-annotated), and the Istio VirtualService/Gateway.

The common/eks-data module reads cluster endpoint / OIDC / CA from the EKS API and is the hub every other module consumes. Per-app state: s3://{bucket}/{octopus.space}/{app}/{module}/terraform.tfstate. (Note: the onboard-app.py tool in iaac-octopus-overrides is the on-prem path, not this one.)

Deploy & upgrade workflow

Every app change flows through the same pipeline. Build is automated; the promotion between environments is a deliberate Octopus deploy.

1 · Commitgit push to the app repo
→
2 · CIGitHub Actions builds & tests, pushes image
→
3 · RegistryECR image (064859874041)
→
4 · Releasemage-runner creates Octopus release w/ new tag
→
5 · DeployOctopus USXpress space — click Deploy (dev→qa→prod)
→
6 · Applymage-runner CD → terraform-variant-apps
→
7 · Liveinfra + Helm chart → pod running

Promotion uses the standard Octopus lifecycle dev → qa → prod (the cloud lifecycle; on-prem uses a separate feature-manual path). The TfDeploy.Apply() pipeline runs infrastructure modules first (replicator → namespace → buckets/kafka/postgres/dynamodb/mongo → role), then the single app module, then any post-deploy script.

The engine — DX & mage-runner

"DX" is the whole deployment platform, not one repo. mage-runner is the Go binary that does the work; DX-Apply is the Octopus step that invokes it; the DX__* library variable sets feed it config.

LayerWhat it is
DX (system)mage-runner + Octopus var sets (DX__Common, DX__EKSCluster, DX__Runner, DX__TFState, DX__AWSAccounts) + DX-Apply + GHA workflows.
DX-ApplyAn Octopus deploy-process step (inline PowerShell) that preps the env and invokes mage.
mage-runnerThe Go binary: reads the spec, picks TF modules, runs them, composes Helm values, talks to Octopus.

The stages — TfDeploy.Apply()

0 · Validateenv + channel guard (feature → dev/qa only)
→
1 · Infrastructurereplicator → buckets → auth → kafka → postgres → dynamodb → mongodb-* → role (IRSA)
→
2 · AppsONE of api/cron/handler/ui → Helm dx-<type>
→
3 · ScriptRunpostStep, outputs.yaml, cleanup

Each infra module runs only if present in the spec — shouldRunTerraform() gates on both spec content and existing S3 state. The namespace is created by mage via the K8s SDK (internal/kube/namespace.go), not by a Terraform module — there is no namespace module.

Where Terraform triggers — TfRun()

The heart of the engine (internal/terraform/terraform_functions.go), per module:

  1. Build path $TERRAFORM_APPS_PATH/modules/<relpath> (e.g. apps/api).
  2. Write *.auto.tfvars.json in the module dir: backend (S3), mage (tags/spec/labels/role_arn/configmaps/aws_secrets), hack (local overrides).
  3. tfexec.NewTerraform(path, terraform) → Init(backend) → Apply → Output.
  4. Cache outputs in GlobalContext.Outputs[module] for later modules.
  5. Persist mage.auto.tfvars.json to S3 (so destroy reuses the same vars).
Key moduleWhat it does
common/eks-data ⭐Reads cluster endpoint / OIDC / CA. The hub every module consumes — if it can't resolve, nothing deploys.
common/rolePer-app IAM role <cluster>-<app> with IRSA trust binding system:serviceaccount:<ns>:<sa> — this is per-app IRSA.
apps/<type>helm_release of dx-{api,cron,handler,ui} from ECR → Deployment, Service, IRSA SA, Istio VirtualService/Gateway.

Onboarding = the first release

Author writes spec.yaml. First Apply() creates the namespace, IAM role + IRSA, any S3/Kafka/PG/Dynamo, and the Helm release.

Upgrade = every release after

Same pipeline, new image tag. Infra modules idempotent; the apps/ module rolls the Deployment. ~200 releases across the fleet.

App chart is dx-{api,cron,handler,ui} (2.x); a legacy variant-* chart is mid-migration and accounts for the handful of failed Helm releases (§10). Repo structure verified against the live mage-runner + terraform-variant-apps repos, 2026-07-15.
